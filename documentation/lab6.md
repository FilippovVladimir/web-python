## 3. Задача 1. Подсчет суммы чисел

### 3.1 Условие

Нужно посчитать сумму чисел от 1 до:

```python
10_000_000_000_000
```

Необходимо реализовать три варианта:

- через потоки;
- через процессы;
- через асинхронность.

## 4. Общая идея решения задачи 1

Диапазон чисел делится на несколько частей. Каждая часть считается отдельно, после чего результаты складываются.

Например:

```text
1 ... N
```

делится на части:

```text
1 ... a
a+1 ... b
b+1 ... c
...
```

В проекте используется переменная:

```python
WORKERS = 8
```

Это значит, что диапазон делится на 8 частей.

## 5. Важное пояснение по сумме

Сумма считается через формулу арифметической прогрессии:

```python
return (end - start + 1) * (start + end) // 2
```

Это математически корректно и позволяет быстро получить результат даже для очень большого числа.

Однако важно понимать: такой способ почти не создает реальной CPU-нагрузки. Поэтому замеры времени в этой реализации больше показывают накладные расходы на запуск потоков, процессов и async-задач, а не реальное ускорение тяжелых вычислений.

Что сказать на защите:

> Для корректного и быстрого подсчета суммы использована формула арифметической прогрессии. В реальной CPU-bound задаче multiprocessing обычно был бы эффективнее threading из-за GIL.



Как работает:

1. Диапазон делится на части.
2. Создается пул процессов.
3. Каждый процесс считает свою часть.
4. Результаты возвращаются в основной процесс.
5. Итоговая сумма получается сложением частичных результатов.

Что сказать:

> В multiprocessing вычисления выполняются в отдельных процессах. Это важно для CPU-bound задач, потому что процессы не ограничены GIL так, как потоки.

## 15. Сравнение подходов

| Подход | Где лучше использовать | Особенность |
|---|---|---|
| Threading | I/O-задачи | удобно для сетевых запросов, но есть GIL |
| Multiprocessing | CPU-задачи | обходит GIL, но требует больше ресурсов |
| Asyncio | I/O-задачи | эффективно при большом количестве сетевых запросов |

## 16. GIL

GIL - это Global Interpreter Lock, механизм CPython, который не позволяет нескольким потокам одновременно выполнять Python-байткод.

Что сказать:

> Из-за GIL потоки не дают настоящего параллелизма для CPU-bound задач. Поэтому для тяжелых вычислений лучше использовать multiprocessing.
